# 全社AI導入 reach→value ギャップマップ(サンプル解説)

全社に AI を配布したあと、「ライセンスは配ったが、業務成果に効いているのか確かめられない」という壁に向き合うためのマップです。**配布した(reach)と効いている(value)の距離**を、成果 ↔ 行動 ↔ 知識・スキルのつながりとして表現しています。

このマップでは、AI を配布された**業務部門の担当者(人事・法務・経理・マーケなど非技術を含む)を「AI enablement サービスの内部顧客」として扱います**。カスタマージャーニーの主人公は業務部門の担当者で、統制設計・enablement・経営はその体験を支える裏方(サービス提供側)に置いています。

- サンプル YAML: [enterprise-ai-reach-value.yaml](./enterprise-ai-reach-value.yaml)
- 開き方: studio の「Open File」でこの YAML を選択します(手順は [samples/README](./README.md))

## このマップで判断できること

| 判断 | マップ上の場所 |
|---|---|
| 配布数を成果と取り違えていないか | Outcome の KPI「value 実証率」(reach 指標と分離)+ 経営・PMO レーンの「reach と value を分けて設計する」 |
| どの業務にユースケースを絞るか | EM の「効くユースケースを平均3.5に絞る」+ enablement レーンの選定タスク |
| 部門ごとにどの統制を当てるか | EM の部門別ガードレール(人事/法務/経理/マーケ)+ 統制設計レーンのタスク |
| 統制だけで成果が出ると誤解していないか | EM の「統制を必要条件として先に整える」の説明文(十分条件はワークフロー再設計) |
| 単一ベンダー標準化の出口を持っているか | EM の「ベンダーロックインの出口を設計する」+ 経営・PMO レーンのタスク |

## 商談・導入検討で 3 分で見る順序

1. **Outcome の KPI**(1 分): 主要 KPI が「value 実証率」であり、配布率・利用率(reach)と切り離して測る設計であることを見ます。「いま何で AI 活用を測っていますか? その指標は配布数と成果を取り違えていませんか?」
2. **CJM の感情曲線**(1 分): 配布直後〜個人試行の谷(使い道が不明・部門規制が曖昧)と、ユースケースを絞ってガードレール内で使う以降の回復を見ます。「いまはこの谷のどこですか?」
3. **EM の部門別ガードレール**(1 分): 人事・法務・経理・マーケで当てる統制が異なることを見ます。「いま一律ルールだけで全部門をカバーしようとしていませんか?」

**自組織版に差し替える項目**: ペルソナ(部門・業務の実態)/ フェーズ名 / SBP のレーン(推進体制の実名)/ KPI の定義と目標値 / 部門別ガードレールの規制と統制例(自社の適用法令に合わせる)。reach≠value の構造・必要条件の並びは多くの組織でそのまま使えます。

## マップの読み方(4 つのエディタ)

### CJM: 業務部門担当者の旅

配布された → 個人で試す → 業務に組み込む → 成果で回る → 職務が変わる の 5 フェーズです。感情曲線が「使い道が不明・部門規制が曖昧」の谷から、ユースケースを絞ってガードレール内で使うことで回復する構造を表します。使わない理由がスキル不足ではなく、使い道の不明さと規制の曖昧さであることを、思考・感情の欄で示しています。

### SBP: 体験を支える 4 つの支援レーン

業務部門(現場)/ 統制設計(法務・情報セキュリティ・IT)/ enablement・推進 / 経営・PMO の 4 レーンが、どの顧客体験をどのタスクで支えるかを接続線で示します。統制設計は「使ってよい状態」を、enablement は「使って効く状態」を作る役割分担です。

### Outcome: 成果と CSF・KPI

KGI「全社に配布した AI を部門横断で成果(value)に化けさせる(配布≠成果を解消)」に対し、CSF(重要成功要因)は「reach 指標と value 指標を分けて設計する」ことです。配布数を成果と取り違えると、必要条件(ワークフロー再設計・職務再定義)への投資が止まるためです。

### EM: 必要な行動と知識・スキル

「必要な行動」に、reach/value の分離・ユースケース集中・ワークフロー再設計・**部門別ガードレール(人事/法務/経理/マーケ)**・統制の必要条件化・職務再定義・ベンダー出口を並べています。画面下部のリソース一覧に、行動ごとのスキル・ナレッジ(独立調査)が並びます。

## reach 指標と value 指標の対比

配布(reach)と成果(value)を別の指標として設計することが出発点です。一次情報(OpenAI の Samsung 展開 blog)では、発表内の定量値はいずれも Codex 全体・韓国全体の数字であり、**Samsung 固有の value 指標はゼロ**でした。

| 種別 | 指標の例 | 意味 |
|---|---|---|
| reach(配布) | ライセンス配布数、WAU(週次アクティブ)、利用率 | 「使ってよい状態」が広がった度合い。成果は保証しない |
| value(成果) | EBIT 寄与、業務時間削減、品質指標、value 実証率 | 「使って効く状態」に届いた度合い。業務そのものの作り直しで決まる |

## 部門別ガードレールのマトリクス

配布範囲が広いほど、部門ごとに固有の規制リスクへ統制を当てる必要があります。一律ルールは部門差を無視すると破綻します。EM の 4 つのガードレール行動の中身です。

| 部門 | 主なリスク | ガードレールの例 |
|---|---|---|
| 人事 | 採用・評価での差別、適用法令 | AI 出力の人手レビュー、選考での自動判定の制限 |
| 法務 | 守秘義務、引用の正確性 | 顧客データの入力制限、出力の弁護士確認 |
| 経理 | 財務統制、監査証跡 | 数値の根拠提示の義務化、承認フローの維持 |
| マーケ | 著作権、ブランド毀損 | 生成物の権利確認、公開前の人手チェック |

## 独立調査の収束数値(横比較はしない)

複数の独立調査が「広く配布されているのに成果到達は低位にとどまる」ことを一貫して示します。ただし**各数値は母集団・指標定義が異なるため、横並びのベンチマークとして比較しないでください**。それぞれ標本・時点・定義を添えます。

| レポート | 標本・時点 | 主な数値 | 指標の定義 |
|---|---|---|---|
| McKinsey「The State of AI in 2025」[二次情報] | n=1,993 / 105カ国、2025-11-05 | EBIT 影響を認めるのは 39%(大半が 5% 未満)、高パフォーマーは約 6% | EBIT への影響認識 |
| BCG「Where's the Value in AI?」[二次情報] | n=1,000 / 59カ国、2024-10-24 | PoC を越える capability 確立は 26%(cutting-edge は 4%) | capability の確立 |
| BCG「AI Radar 2025」[二次情報] | n=1,803、2025-01 | significant value の実現は約 25% | significant value の実現 |

定義が異なるため単純比較はできませんが、「配布が広くても成果到達は低位に収束する」という傾向は共通しています。

## 反証と留保

このマップを使うときに注意する留保条件です。

- **MIT NANDA「GenAI パイロットの約 95% が ROI 不発」[要再検証]**: 標本数が報道と原典で食い違い(150/350 と 52/153 など)、査読前で、利益相反の指摘もあります。裸で断定せず傾向の傍証としてのみ扱うべき数値です。このためサンプルの YAML では引用リンク(url)ノードにしていません。
- **統制は必要条件だが十分条件ではない**: MIT の研究は成果不発の主因を統制不足でなくワークフロー統合と組織学習の欠如に求めています。統制は「使ってよい状態」を作りますが、「使って効く状態」までは作りません。
- **職務再定義の負の側面**: 過剰な置換は品質低下・顧客離反を招くことがあります(Klarna が約700名分の業務を AI 置換後に一部再雇用した例)。段階的に進めます。
